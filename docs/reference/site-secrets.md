# Site secrets

Up stores per-site secrets as AES-256-GCM ciphertext in a dedicated `SiteSecrets` Durable Object. The encryption key is generated by `bun run setup`, stored locally with mode `600`, and uploaded as a Worker secret. It is never written to Wrangler vars, R2, the registry, deployment manifests, receipts, or git.

Only site owners and Up administrators can create, list, or delete secret capabilities. List responses contain names, allowed hosts, and update times—never ciphertext or plaintext.

Dynamic code receives a site-specific `UP_SECRETS` stub. It cannot read a secret. Instead, it asks the stub to make an allowlisted HTTPS request:

```js
const response = await env.UP_SECRETS.fetch(
  'https://secrets.internal/use/API_TOKEN',
  {
    method: 'POST',
    headers: { 'content-type': 'application/json' },
    body: JSON.stringify({
      url: 'https://api.example.com/jobs',
      method: 'POST',
      body: JSON.stringify({ action: 'run' }),
    }),
  },
);
```

The secret is injected as a bearer credential by the trusted Durable Object. Dynamic code cannot override authorization, cookies, host, or Access headers. Outbound hosts are exact allowlist entries. Only bounded text/JSON responses are returned, and direct occurrences of the secret are redacted.
